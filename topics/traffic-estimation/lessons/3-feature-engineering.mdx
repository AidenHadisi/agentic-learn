# Feature Engineering

<div className="lesson-meta">Turn raw data sources into a feature vector you can feed into a model.</div>

You know what data sources exist. Now the question is: how do you turn "a CrUX rank bucket" and "an ads.txt file" into numbers a model can learn from? This section covers the feature vector design, transformations, the domain-matching problem, and the collection pipeline.

## The Feature Vector

Here's every feature worth extracting, organized by source:

### Popularity ranks (the heavy hitters)

| Feature | Source | Type | Encoding |
|---------|--------|------|----------|
| `crux_in` | CrUX | binary | 1 if origin in CrUX, 0 if absent |
| `crux_log_rank` | CrUX | float | `log10(rank_bucket)`, sentinel for absent |
| `crux_rank_ord` | CrUX | ordinal | bucket index 0..N |
| `tranco_in` | Tranco | binary | present in top-1M list |

| `cf_in` | Radar | binary | in any Radar bucket |
| `cf_log_bucket` | Radar | float | `log10(bucket)` |

| Feature | Source | Type | Encoding |
|---------|--------|------|----------|

| `domain_is_new` | RDAP | binary | registered within last 365 days |

That's roughly **30-35 features** before interactions — a good starting size for a model with thousands of labeled domains.

## Why Log Everything

Website traffic follows a **power law**. The difference between rank 1 and rank 100 is enormous; the difference between rank 500,000 and rank 500,100 is negligible. Raw ranks and traffic numbers span many orders of magnitude, which breaks linear models and makes tree splits inefficient.

The relationship between rank and traffic is itself a power law. Research on CrUX data found:

`yearly_pageviews ≈ 4 × 10¹² × rank^(-1.01)`

So `log(traffic) ≈ a + b × log(rank)` — log-rank is the natural feature for predicting log-visits.

<Callout variant="key">
Always predict `log1p(monthly_visits)` as your target variable, not raw visits. Back-transform with `expm1()` for the final estimate. This makes the error metric meaningful across traffic tiers.
</Callout>

## Handling Missing Data

Not every domain appears in every source. A site missing from CrUX might have 4,000 monthly visits (just below the threshold) or 40. This is **informative missingness** — the absence itself is a feature.

<Tabs>
	<Tab label="The Pattern">
		For every ranked source, create two features:

		1. **Binary presence flag** (`crux_in`, `tranco_in`, etc.)
		2. **Log-rank with sentinel for missing** — use a value beyond the list's range

		```python
		CRUX_ABSENT = 1e8      # beyond 18M CrUX origins
		TRANCO_ABSENT = 2e6    # beyond 1M Tranco list
		UMBRELLA_ABSENT = 2e6  # beyond 1M Umbrella list

		def encode_rank(rank, absent_sentinel):
		    if rank is None:
		        return 0, np.log10(absent_sentinel)  # (in_flag, log_rank)
		    return 1, np.log10(max(rank, 1))
		```

		This lets the model learn different behaviors for "ranked low" vs "not ranked at all."
	</Tab>
	<Tab label="What NOT to do">
		**Don't impute the mean rank.** If the average Tranco rank in your training set is 250,000, filling missing domains with 250,000 makes them look like mid-tier sites — biasing the model toward average predictions for unknown domains.

		**Don't use zero.** Zero is ambiguous — does it mean "rank 0" (doesn't exist) or "missing"? Use explicit flags.

		**Don't drop rows with missing features.** Long-tail domains will be missing from most sources. Dropping them throws away exactly the cases you need to handle.
	</Tab>
</Tabs>

For count features (keyword count, ads.txt lines), distinguish between **truly zero** and **crawl failure**:

<CodeBlock language="python" title="Distinguishing zero from missing" code={`def ads_features(domain, crawl_result):

## The eTLD+1 Problem

Every source uses a different domain format. CrUX uses full origins (`https://www.example.com`), Tranco uses hostnames (`example.com`), Umbrella includes subdomains (`mail.google.com`), and ads.txt lives at the apex domain. You need a canonical key to join them.

The **eTLD+1** (effective top-level domain + 1) is the standard:
- `www.example.com` → `example.com`
- `blog.example.co.uk` → `example.co.uk`
- `user.github.io` → `user.github.io` (github.io is a public suffix)

<CodeBlock language="python" title="Domain normalization with tldextract" code={`import tldextract
from urllib.parse import urlparse

extract = tldextract.TLDExtract(include_psl_private_domains=False)

def to_etld1(value: str) -> str:
    """Normalize any domain/URL to its eTLD+1."""
    host = urlparse(value).hostname if "://" in value else value
    host = (host or value).lower().strip(".").removeprefix("www.")
    parts = extract(host)
    return parts.top_domain_under_public_suffix

# Examples:
# to_etld1("https://www.bbc.co.uk/news")  → "bbc.co.uk"
# to_etld1("mail.google.com")              → "google.com"
# to_etld1("example.com")                  → "example.com"`} />

<Callout variant="warn">
Never roll your own domain parser with string splitting. `".".join(parts[-2:])` turns `bbc.co.uk` into `co.uk`. Always use the Public Suffix List via `tldextract` or equivalent.
</Callout>

**Join strategy per source:**

| Source | Raw key format | Join rule |
|--------|---------------|-----------|
| CrUX | `https://www.example.com` | Parse origin → hostname → eTLD+1 |
| Tranco | `example.com` | Already close; apply eTLD+1 to be safe |
| Radar | ~eTLD+1 already | Direct join |
| Umbrella | `mail.google.com`, `www.google.com` | Roll up: best (min) rank among all hosts under eTLD+1 |
| DataForSEO | `example.com` | eTLD+1 |
| ads.txt | Fetched from eTLD+1 | Direct |

For Umbrella, aggregate subdomains:

<CodeBlock language="python" title="Aggregating Umbrella subdomains" code={`umbrella_by_site = (
    df.assign(site=df.host.map(to_etld1))
      .groupby("site")
      .agg(
          umbrella_best_rank=("rank", "min"),
          umbrella_host_count=("host", "nunique")
      )
)`} />

### Rate limiting

<CodeBlock language="python" title="Polite crawling with backoff" code={`import random, time, requests

def fetch_with_backoff(url, session, max_retries=3, timeout=10):
    for attempt in range(max_retries):
        try:
            r = session.get(url, timeout=timeout,
                           headers={"User-Agent": "TrafficBot/1.0"})
            if r.status_code == 429:
                wait = float(r.headers.get("Retry-After", 5))
                time.sleep(wait)
                continue
            return r
        except requests.RequestException:
            time.sleep(random.uniform(0, min(60, 2 ** attempt)))
    return None`} />

### Multicollinearity between rank features

CrUX, Tranco, and Umbrella are correlated — they're all measuring some flavor of "popularity." For tree models (XGBoost, LightGBM), this is fine for accuracy but hurts interpretability. For linear models, it's a problem.

**Practical approaches:**

<Tabs>
	<Tab label="For trees (recommended)">
		Keep all rank features. Trees will split on whichever provides the most information gain at each node. The redundancy doesn't hurt prediction — it provides robustness when one source is missing.

		Prune later with permutation importance if you need to reduce serving cost.
	</Tab>
	<Tab label="For linear models">
		Collapse correlated ranks into a single "popularity score":

		```python
		import numpy as np

		pop_score = np.nanmean([
		    crux_log_rank,
		    tranco_log_rank,
		    cf_log_bucket,
		    umbrella_log_rank,
		], axis=0)

		n_sources_present = crux_in + tranco_in + cf_in + umbrella_in
		```

		Or use PCA on the rank features and keep the first 1-2 components.
	</Tab>
</Tabs>

### How many features is enough?

With ~5,000-50,000 labeled domains, **20-25 well-chosen features** is the sweet spot. Research shows diminishing returns beyond ~30 features for this kind of domain-level prediction. Start with all ~35, then prune:

<Stepper>
	<Step>Train a baseline model with all features.</Step>
	<Step>Run permutation importance on the validation set — this measures how much accuracy drops when you shuffle each feature.</Step>
	<Step>Drop features with near-zero permutation importance.</Step>
	<Step>Check for VIF (variance inflation factor) among remaining features — drop one from any pair with VIF above 10.</Step>
	<Step>Retrain and verify no significant accuracy loss.</Step>
</Stepper>

<Callout variant="tip">
Watch for overfitting on rare features. Binary flags like "has SSP X" might only be true for 50 domains in your training set — the model can memorize these. Prefer features with broad support across your labeled data.
</Callout>
